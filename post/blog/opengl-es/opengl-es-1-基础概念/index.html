OpenGL ES 全称 OpenGL for Embedded Systems，是专门为嵌入式设备改良的 开源图形库

OpenGL ES 是 OpenGL 的简化版本，简化了一些功能。

OpenGL ES 的实现是基于 C 的

一、渲染上下文 EAGLContext

实际上，OpenGL 是一个 状态机，管理着若干的 状态变量 （如 深度测试状态 GL_DEPTH_TEST）。我们可以通过其 api 对这些状态进行管理 & 获取，而每个状态都有 默认值。最终我们会根据这些 状态变量 进行 渲染。

而这些渲染所需的 状态变量，都保存到 渲染上下文（context） 中。

对于 iOS 平台而言，我们通过 EAGLContext 来管理我们的渲染上下文。

并且在每一个线程当中，都会维护一个上下文

下面例子可以证明：

    _ctx = [[EAGLContext alloc] initWithAPI: kEAGLRenderingAPIOpenGLES3];
    [EAGLContext setCurrentContext: _ctx];
    
    NSLog(@"CTX0： %@ - %@", [NSThread currentThread], [EAGLContext currentContext]);
    
    dispatch_async(dispatch_get_global_queue(0, 0), ^{
        NSLog(@"CTX1：%@ - %@", [NSThread currentThread], [EAGLContext currentContext]);
    });

结果输出为：

2020-05-07 01:24:18.684586+0800 OpenGLTest[3043:179547] CTX0： <NSThread: 0x600001db4100>{number = 1, name = main} - <EAGLContext: 0x6000008b8220>
2020-05-07 01:24:18.684762+0800 OpenGLTest[3043:179720] CTX1：<NSThread: 0x600001d2c640>{number = 6, name = (null)} - (null)

相关 api：

  • - (nullable instancetype) initWithAPI:(EAGLRenderingAPI) api; 创建上下文，有可能会创建失败。比如某些系统不支持 kEAGLRenderingAPIOpenGLES3 就会返回 nil

  • + (BOOL) setCurrentContext:(nullable EAGLContext*) context; 设置当前线程的上下文

  • + (nullable EAGLContext*) currentContext; 获取当前线程的上下文，未设置的时候会返回 nil

二、EGL （EAGL）

EGL 是 OpenGL ES 的上屏框架，而 Apple 使用了自己实现的 EAGL，基于 Core Animation 框架来控制显示。

EGL 允许我们选择 屏上渲染 和 离屏渲染，前者是直接将结果渲染到屏幕上，而后者是通过将内容渲染到 buffer 中，然后再展示。

对于离屏渲染，有些实现是通过 SwapBuffer 来实现，简单来说就是一个 buffer 用来显示，另一个用来后台渲染，渲染完成后交换两个 buffer。而 iOS 中的实现则是通过 context 的 presentRenderBuffer: api 结合系统内部实现的。

内容展示 CAEAGLayer

继承自 CALayer，用以展示 renderBuffer。

通过调用 EAGLContext 的 - (BOOL)renderbufferStorage:(NSUInteger)target fromDrawable:(nullable id<EAGLDrawable>)drawable; 函数，可以将 buffer 与 layer 绑定到一起：

[_ctx renderbufferStorage: GL_RENDERBUFFER fromDrawable: _glLayer];

注意：

  • CALayer 的 Opaque 默认是 false，即透明的，需要设置为 true 才能展示内容

  • drawableProperties 是其子类唯一的属性，保存这个 layer 的配置 dict，有如下两个 key：

    • kEAGLDrawablePropertyRetainedBacking：控制调用 presentRenderbuffer 后是否对 layer 的 content 进行 retain

    • kEAGLDrawablePropertyColorFormat：颜色的样式，一般用 kEAGLColorFormatRGBA8 值

PS:

如果想让自定义的 UIView 的 layer 使用 CAEAGLayer，重写其 + (Class)layerClass 方法即可：

+ (Class)layerClass {
    return [CAEAGLLayer class];
}

三、RBO & FBO 离屏渲染

1. RBO 渲染缓存区 （Render Buffer Object）

用以将 储存渲染的内容，有可能是 颜色（color）、深度（depth）、模板（stencil）。

首先需要 创建和绑定

    // 创建 Render Buffer Object
    glGenRenderbuffers(1, &_RBO);
    glBindRenderbuffer(GL_RENDERBUFFER, _RBO);

然后再为其指定储存区 （调用上述的 - (BOOL)renderbufferStorage:(NSUInteger)target fromDrawable:(nullable id<EAGLDrawable>)drawable; 方法），这样就可以将 CAEAGLayer 的 contents 作为 Render Buffer 的储存区

[_ctx renderbufferStorage: GL_RENDERBUFFER fromDrawable: _glLayer];

2. FBO 帧缓冲区 （Frame Buffer Object）

FBO 只是充当管理角色，负责管理 RBO。

FBO 不含储存区，可以将含有储存区的 RBO 附着（attach）到 FBO 上做管理

同样需要先 创建和绑定

    // 创建 Frame Buffer Object
    glGenFramebuffers(1, &_FBO);
    glBindFramebuffer(GL_FRAMEBUFFER, _FBO);

然后将附着 RBO 到 Color Attachment （或者 Depth Attachment、Stencil Attachment）上去：

glFramebufferRenderbuffer(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT0, GL_RENDERBUFFER, _RBO);

三、着色程序 Program

OpenGL 2.0 的最大特性是 可编程管线，意味着我们可以实现自己的着色器，包括 顶点着色器 & 片元着色器。

GLProgram 用来管理这两种着色器管线，使用一个着色程序需要下列步骤：

    1. 创建 GLuint glCreateProgram ()

    此处会返回句柄

    1. 附着 着色器（source） glAttachShader (GLuint program, GLuint shader)

    第一个参数是前一步的 着色程序 句柄，第二个是 着色器 句柄（下面会提到）

    1. 链接 glLinkProgram (GLuint program)
    1. 使用 glUseProgram (GLuint program)

第二步的着色器（source）哪里来的呢

由于是可编程的，所以我们需要去自行构造和编译，有如下步骤

    1. 创建 source GLuint glCreateShader (GLenum type)

    此处的 type 有两种：顶点着色器 （GL_VERTEX_SHADER） & 片元着色器 （GL_FRAGMENT_SHADER）

    然后返回 着色器 句柄

    1. 载入 着色器 内容 glShaderSource (GLuint shader, GLsizei count, const GLchar* const *string, const GLint* length)

    • shader 着色器句柄

    • count 着色器个数，一般为 1，也可以多个同时读取

    • string 着色器字符

    • length 长度数组，只有一个的时候传入 NULL 即可，多个着色器可以传入一个数组

    1. 编译 着色器 glCompileShader (GLuint shader)

四、顶点绘制 （VAO & VBO & EBO）

  • VBO - Vertex Buffer Object 顶点缓存对象，管理一段顶点数据

  • VAO - Vertex Array Object 顶点数组对象，管理着众多的 VBO。 仅 OpenGL ES 3.0 之后支持

  • EBO - Element Buffer Object 索引（index）缓存对象，可以对顶点进行复用，提高性能

顶点数据提交方式

CPU 在提交顶点数据的时候，有如下方式：

    1. 通过 CPU 的内存复制过去
    1. 通过 预留一段显存储存区（buffer），每次同时提交。这种就需要 VBO

显然，方法 1 是存在性能瓶颈的，方法 2 会更加方便和快速
