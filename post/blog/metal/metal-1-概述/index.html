Metal 是苹果推出专用的与 GPU 沟通的框架，类似于 OpenGL ES，但是 OpenGL ES 苹果明确说明在 iOS 12 已经被弃用了，虽然现在还能使用，但是不再为其进行更新了，也指不定哪个新系统会直接不让使用了。

因此，现在学习 Metal，对于 iOS 音视频方面开发有一定的前瞻性。

一、Metal VS OpenGL ES

Metal 比 OpenGL ES 的优点：

  • 1、Metal Api 更加 人性化，相比于 OpenGL 使用句柄的操作以及 C 语言的 Api，Metal 是 面向对象 的；

  • 2、Metal 的 多线程支持，相比于 OpenGL 的上下文与线程的绑定，Metal 是 线程安全 的；

  • 3、Metal 的 性能 优于 OpenGL，有如下方面：

    • a、Metal 的 GPU 指令是在 创建的时候就确定 的，因此不像 OpenGL 一样，需要在渲染的时候做状态检查；

    • b、Metal 的 shaders 在程序编译的时候就编译好了，而 OpenGL 的 shaders 是在运行的时候才进行编译的；

    • c、Metal 对 GPU 的指令有底层优化（因为是 Apple 推出的）；

  • 4、Metal 支持 Compute shaders，能让我们使用 GPU 来做一些更加复杂的 并行计算 任务，如机器学习等，而 OpenGL 几乎只能支持与图形图像相关的。

Metal 比 OpenGL ES 的缺点：

  • 1、Metal 仅支持 Apple 平台，而 OpenGL 是跨平台的；

  • 2、Metal 对旧设备与系统支持不友好，而由于 OpenGL “出生” 较早，因此对旧的支持更好；

二、Metal 框架概览

Metal 和 OpenGL ES 一样，使用的是 Client - Server 模型：

  • Client 我们的 App

  • Request 一系列的 GPU 指令（Commands）

  • Server 设备的 GPU

  • Response 设备的 GPU 工作的响应（成功、失败、状态等）

我们的 App 通过提交 Commands 到 “公有” 的设备 GPU，最终将完成的任务通知到提交者，即我们的 App。

具体的流程如下：

  • Function 函数，也就是我们的顶点函数（Vertex Function）& 片元函数（Fragment Function）& 计算函数（Kernel Function），对应的类是 MTLFunction，最终会被组装到 MTLLibrary 中

  • Resource 资源，GPU 可访问的容器，用以 管理 & 储存 我们的数据，包括有缓冲区（MTLBuffer）& 纹理（MTLTexture）

  • Commands 指令，对 GPU 进行操作（包括组装、提交、执行）的指令集合，所有对 GPU 的操作和资源都会被封装到指令编码器（MTLCommandEncoder）中，然后会将他们通过指令缓冲区（MTLCommandBuffer）送到 GPU 中执行，GPU 的执行顺序由指令队列（MTLCommandQueue）进行管理

这些只是一个概述，具体的使用方式，还是要从后面的 demo 中来体会。

三、Metal 常用的类

在 OC 中，Metal 是基于协议（protocol-base）的，因此大多数我们在定义对象的时候需要 id<MTLXXXXX> 的方式。

1、设备相关

  • MTLDevice GPU 设备

    在 iOS 中，没有外接 GPU（External GPU），一般获取默认的 GPU 即可，通过 MTLCreateSystemDefaultDevice() 方法。

2、指令相关

  • MTLCommandQueue 组织 GPU 执行指令的 队列

  • MTLCommandBuffer 一个储存编码后（encoded）的 GPU 执行指令的 容器

  • MTLCommandEncoder 将一系列 GPU 指令写入指令缓冲区（Command Buffer）的 编码器 包含如下子协议：

    协议 Protocol对应任务 Task
    MTLRenderCommandEncoder图形渲染
    MTLComputeCommandEncoderGPU 计算
    MTLBlitCommandEncoder内存管理，包括拷贝、过滤、填充
    MTLParallelRenderCommandEncoder多图形并行渲染
    MTLResourceStateCommandEncoder修改资源配置状态

    encoder 可以 通过对应的 descriptor 来创建，然后可以写入的指令包括有：

    • 设置对应的 pipelineState

    • 设置资源，如 buffer、texture

    • 绘制图元指令 drawPrimitives

    • 编码完成 endEncoding

    • 等等….

3、渲染相关

  • MTLRenderPipelineState 一个包括在渲染过程中使用的 渲染函数（Rendering Function）和配置状态（Configuration State） 的对象

  • MTLRenderPipelineDescriptor 渲染过程中使用到的参数的 管道（pipeline）描述对象，包括有光栅化参数（比如多重采样）、可见性（裁剪）、混合、曲面细分（tessellation）和 图形函数（graphics function） 等。这个描述对象后续会用来创建渲染状态（MTLRenderPipelineState）。

  • MTLRenderPassDescriptor 渲染过程的最后，渲染结果（Render Target） 的描述对象。上面会包括有各种附着点（Attachment），包括颜色（colorAttachments）& 深度（depthAttachment）& 模板（stencilAttachment）三大附着点。

另外还有一些深度测试、模板测试相关的类，可以自行去了解

4、并行计算相关

  • MTLComputePipelineState 并行计算管道 状态对象，是一个完全线程安全的对象，可以同时被应用到多个 MTLComputeCommandEncoder 中

  • MTLComputePipelineDescriptor 一个 并行计算的 描述对象，会指定 计算函数（computeFunction 属性，是一个 kernel function）等。后续会用来创建 MTLComputePipelineState 对象

  • MTKTextureLoader 用以将图片解码成 Metal 纹理的 纹理加载器

5、函数 Function 相关

  • MTLLibrary Metal 函数（MTLFunction）的集合

  • MTLFunction Metal 函数。包括顶点函数（Vertex Function）& 片元函数（Fragment Function）& 计算函数（Kernel Function）

6、资源

  • MTLResource 一个 可被 GPU 访问的内存，只能使用 MTLDevice 来创建。

    包括以下的子协议：

    • MTLBuffer 保存我们 自定义数据结构 的资源

    • MTLTexture 保存 格式化后的图片数据（data） 的资源

  • MTLTextureDescriptor 纹理描述对象，用来 配置我们 GPU 纹理数据。该对象可以被复用，对这个对象的修改不会影响已经使用该对象创建的图片数据

7、展示相关

  • MTLDrawable 用来 展示（present） 渲染结果或者写入的结果的对象

    官方定义如下：

    @protocol CAMetalDrawable <MTLDrawable>

/* This is an object that conforms to the MTLTexture protocol and will
* typically be used to create an MTLRenderTargetDescriptor. */

@property(readonly) id<MTLTexture> texture;

/* This is the CAMetalLayer responsible for displaying the drawable */

@property(readonly) CAMetalLayer *layer;

    包含一个 texture 用来被渲染或者写入，以及一个 layer 用来展示内容。

四、Metal 实例关系

  • 一个 Queue 包含多个 Buffer

  • 一个 Buffer 包含多个 Encoder

  • 一个 Encoder 可能包含 多个不同 的 State（比如 Render State，depth stencil state…），以及需要被使用的 Resources

另：

  • 一个 State 是由一个 Descriptor 组成的，一个 Descriptor 中会包含有一个 Library

  • 一个 Library 中包含多个 Function
