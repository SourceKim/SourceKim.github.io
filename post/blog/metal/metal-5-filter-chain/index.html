Metal - 5 滤镜链（渲染到纹理）

本文要实现的效果是对一张图片同时使用灰度滤镜以及亮度滤镜，使用 滤镜链 实现，最终把应用滤镜链的结果输出到 Layer 上，并且可以调节灰度和亮度程度之后实时渲染显示。

Metal 中要实现滤镜链，需要将 中间滤镜的渲染结果临时保存到一个纹理中，最后一个滤镜的渲染结果直接保存到 Drawable 的纹理中。因此需要用到 渲染到纹理 的技术。

在本文中中间滤镜仅一个 - 灰度滤镜，最后的滤镜是亮度滤镜。

同时因为 Metal 的 标准坐标系 NDC（Normalized Device Coordinate）、 帧缓存坐标系 FrameBuffer Coordinate （也叫 Viewport Coordinate）以及纹理坐标系（Texture Coordinate）的原点不一致，还会分别对他们进行介绍和解析。

具体代码可以访问 仓库 的 4. 滤镜链。

最终效果：

一、渲染到纹理

在本 demo 中，灰度滤镜是中间滤镜，因此该次渲染的渲染结果会被保存到纹理中。

在 Metal 中要渲染到纹理有如下步骤：

  • 1、创建空纹理

  • 2、在渲染时候将空纹理设值到 RenderTarget 的描述对象（MTLRenderPassDescriptor）的 ColorAttachment0 的 texture 中

  • 3、提交指令，执行渲染

1、空纹理的创建

为了方便使用，我将其封装到了 MetalUtils 中了：

+ (nullable id<MTLTexture>)createEmptyTexture: (id<MTLDevice>)device
                                    WithWidth: (size_t)width
                                   withHeight: (size_t)height
                                        usage: (MTLTextureUsage)usage {
    
    MTLTextureDescriptor *texDescriptor = [MTLTextureDescriptor texture2DDescriptorWithPixelFormat: MTLPixelFormatRGBA8Unorm width: width height: height mipmapped: false];
    texDescriptor.usage = usage;
    
    id<MTLTexture> texture = [device newTextureWithDescriptor: texDescriptor];
    
    return texture;
}

共 4 个参数：

  • device 设备，Metal 的 资源创建依赖 Device

  • width 纹理宽度

  • height 纹理高度

  • usage 纹理的用处，是 MTLTextureUsage 枚举，有如下枚举值：

    • MTLTextureUsageUnknown 未知用处（运行时设置），会损耗性能
    • MTLTextureUsageShaderRead 该纹理将被读取
    • MTLTextureUsageShaderWrite 该纹理将被写入
    • MTLTextureUsageRenderTarget 该纹理将作为渲染目标
    • MTLTextureUsagePixelFormatView 在 PixelFormatView 被使用（暂不介绍）

回到 demo 中，这个中间 buffer 是 当次渲染的渲染目标 ，也将 作为下一次渲染的输入被读取（采样），因此 usage 是 MTLTextureUsageShaderRead | MTLTextureUsageRenderTarget。

完整调用：

    _grayResultTexutre = [MetalUtils createEmptyTexture: _device
                                              WithWidth: CGImageGetWidth(cgImage)
                                             withHeight: CGImageGetHeight(cgImage)
                                                  usage: MTLTextureUsageShaderRead | MTLTextureUsageRenderTarget];

2、空纹理关联到渲染目标

以往为了要将渲染结果展示，我们的渲染目标是与 Layer 关联起来的：

    RenderTargetDesc.colorAttachments[0].texture = currentDrawable.texture;

但是这次我们需要渲染到纹理上，因此要把 渲染目标和要渲染到的纹理 关联起来：

    _grayRenderTargetDesc.colorAttachments[0].texture = _grayResultTexutre;

其中 _grayResultTexutre 就是第 1 步创建的空纹理。

3、提交指令，执行渲染

指令是被 ”装“ 到 Encoder 上的，因此渲染过程也大同小异，但是由于 不需要展示渲染结果（即不需要上屏 swap buffer），所以不需要 presentDrawable。

渲染代码如下：

- (void)render_gray {
    
    // 将本次 Command Encoder 和 渲染的目标（MTLTexture）关联起来
    _grayRenderTargetDesc.colorAttachments[0].texture = _grayResultTexutre;
    
    id<MTLCommandBuffer> commandBuffer = [_queue commandBuffer];
    commandBuffer.label = @"Gray Command Buffer";
    
    id<MTLRenderCommandEncoder> encoder = [commandBuffer renderCommandEncoderWithDescriptor: _grayRenderTargetDesc];
    encoder.label = @"Gray Command Encoder";
    
    [encoder setViewport: (MTLViewport) {
        .originX = 0,
        .originY = 0,
        .width = _sourceTexutre.width,
        .height = _sourceTexutre.height,
        .znear = 0,
        .zfar = 1
    }];
    
    [encoder setRenderPipelineState: _grayRenderPipelineState];
    
    [encoder setVertexBytes: grayVertices
                     length: sizeof(grayVertices)
                    atIndex: 0];
    
    [encoder setVertexBytes: texCoor
                     length: sizeof(texCoor)
                    atIndex: 1];
    
    [encoder setFragmentTexture: _sourceTexutre
                        atIndex: 0];
    
    [encoder setFragmentBytes: &_grayIntensity
                       length: sizeof(float)
                      atIndex: 0];
    
    [encoder drawIndexedPrimitives: MTLPrimitiveTypeTriangleStrip
                        indexCount: 6
                         indexType: MTLIndexTypeUInt32
                       indexBuffer: _indexBuffer
                 indexBufferOffset: 0];
    
    [encoder endEncoding];
    
    [commandBuffer commit];
}

注意，Command Buffer 并不需要 presentDrawable:。

二、渲染流程的注意点

先看看渲染流程：

一共有两个执行两次 Draw Call，即两个 Command Buffer 的提交：

  • 第一次是 gray 滤镜，将输入的原图经过灰度处理渲染到纹理上

  • 第二次是 brightness 滤镜，将 gray 输出的纹理作为输入，经过亮度处理渲染到 Layer 的纹理上

有如下需要注意的地方：

  • 1、两个渲染目标的 texture 的 pixelformat 是不一样的

  • 2、两个渲染目标的 texture 的 Dimentions 即 size 是不一样的

  • 3、两次渲染的顶点不一样的 （坐标系的不一样）

1、渲染目标的 PixelFormat 不一样

为了在 shader 中的 rgba 和输入的 rgba 能保持一致，在 MetalUtils 中进行纹理创建和加载都使用了 MTLPixelFormatRGBA8Unorm，但是 Layer 使用的是 MTLPixelFormatBGRA8Unorm。因此，在设置 RenderTarget（MTLRenderPassDescriptor）的 ColorAttachment 的 PixelFormat 的时候要注意使用对应的 PixelFormat，否则会出现 图像颜色变蓝。

因此：

  • 对于 gray 滤镜的 PixelFormat 要使用和 texture 的一样，使用 MTLPixelFormatRGBA8Unorm

  • 对于 brightness 滤镜的 PixelFormat 要使用和 layer 的一样，使用 MTLPixelFormatBGRA8Unorm

2、渲染目标的 Dimentions 即 size 不一样

对于滤镜链而言，我们希望在处理的过程中，如非会改变到纹理尺寸的 filter，都需要保持尺寸一致，而对于最后的输出而言，我们希望输出的纹理大小和 Layer 的大小一样 （一般会通过顶点控制使滤镜处理后的纹理居中于最后 Layer 的纹理）。

所以 size 是不一样的：

  • Gray 滤镜渲染目标的 texture 大小是输入图片的大小 548x568

    在 createEmptyTexture: WithWidth: withHeight: usage: 传入输入图片的宽高

  • Brightness 滤镜渲染目标的 texture 大小是输出 Layer 的大小 414x874

    这个是自动指定的。

另外，在 Encoder 设置 Viewport 的时候也需要使用不同的大小：

Gray：

    - (void)render_gray {

        // .....
        
        [encoder setViewport: (MTLViewport) {
            .originX = 0,
            .originY = 0,
            .width = _sourceTexutre.width,
            .height = _sourceTexutre.height,
            .znear = 0,
            .zfar = 1
        }];

        // .....
    }

Brightness：

    - (void)render_brightness {
        
        // .....
        
        [encoder setViewport: (MTLViewport) {
            .originX = 0,
            .originY = 0,
            .width = _layer.drawableSize.width,
            .height = _layer.drawableSize.height,
            .znear = 0,
            .zfar = 1
        }];

        // .....
    }

3、渲染顶点不一样

两次渲染使用的纹理坐标是一样的：

static const float texCoor[] = {
    0, 0, // 左下角
    1, 0, // 右下角
    0, 1, // 左上角
    1, 1, // 右上角
};

但是使用的顶点是却是不一样的：

static const float grayVertices[] = {
    -1, 1, 0, 1, // 左上角
    1, 1, 0, 1, // 右上角
    -1, -1, 0, 1, // 左下角
    1, -1, 0, 1, // 右下角
};

static const float brightnessVertices[] = {
    -0.5, -0.5, 0, 1, // 左下角
    0.5, -0.5, 0, 1, // 右下角
    -0.5, 0.5, 0, 1, // 左上角
    0.5, 0.5, 0, 1, // 右上角
};

不一样的地方主要有两处：

  • 1、为了让中间滤镜即 Gray 滤镜的 渲染结果和输入纹理一样，顶点 x 和 y 使用了 -1 和 1；而为了让最后输出的结果只占整体纹理的一半，顶点使用了 -0.5 和 0.5，如果要保持比例的话，也应该是一个小数，此处只是举例子。

  • 2、为了 防止纹理上下颠倒，滤镜链中非最后一个滤镜的顶点都应该需要和纹理坐标上下相反。（即顶点的左上角的位置和纹理的左下角对应）

那么为什么需要相反呢？

Metal 渲染的时候有 3 种坐标系：

  • a、设备归一化坐标系 NDC (Normalized Device Coordinate)

    我们在 构造顶点 的时候，使用的是该坐标系

    该坐标系范围是 [-1, 1]，原点在中心，x 轴从左向右，y 轴从下到上，z 轴从人指向设备。

  • b、帧缓存坐标系 Framebuffer Coordinate (Viewport coordinate)

    当我们渲染到 RenderTarget 的 ColorAttachment，我们使用的是该坐标系

    该坐标系的范围是 Viewport 的范围，原点（0，0）在左上角 y 轴从上到下

  • c、纹理坐标系 Texture Coordinate

    在我们需要对纹理进行采样的时候，使用的是纹理坐标系

    该坐标系的范围是 [0, 1]，原点（0，0）在左上角 y 轴从上到下

在我们绘制图片的时候，通常看到的纹理都是 ”上下颠倒“ 的，只是在最后绘制到 FrameBuffer 的时候由于坐标系的 Y 与我们看到的是上下相反的，所以又 ”调转“ 了。

但是在实现滤镜链的时候，我们希望纹理在链中传递的时候是保持 ”上下颠倒“ 的，这样在最后才能再次翻转成正面。
