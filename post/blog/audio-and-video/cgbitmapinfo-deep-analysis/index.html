CGBitmapInfo / CGImageAlphaInfo / CGImageByteOrderInfo 详解

 发表于 June 14, 2020  |  3 分钟  |  466 个字  |  Kim
  • CGImageAlphaInfo
  • CGImageByteOrderInfo
  • 总结

在操作 CGImage 或者 CGContext 的时候，经常会传入一个 Info 去描述颜色、透明通道的排列信息。

先说结论：在 iOS 中，颜色排序和对应的 BitmapInfo 如下。

  • R G B A – kCGImageAlphaPremultipliedLast | kCGImageByteOrder32Big

  • A R G B – kCGImageAlphaPremultipliedFirst | kCGImageByteOrder32Big

  • B G R A – kCGImageAlphaNoneSkipFirst | kCGImageByteOrder32Little

  • A B G R – kCGImageAlphaPremultipliedLast | kCGImageByteOrder32Little

  • CMYK – kCGImageAlphaNone (or 0)

  • Gray – kCGImageAlphaNone (or 0)

另外还会顺便介绍 预乘 & 大小端序。

CGBitmapInfo 的定义如下

typedef enum CGBitmapInfo : uint32_t {
    ...
} CGBitmapInfo;

具体内容不关心，是一个 UInt32 的枚举。

在我们使用的时候，比如说创建 CGImageRef 的时候

CGImageRef __nullable CGImageCreate(size_t width, size_t height,
    size_t bitsPerComponent, size_t bitsPerPixel, size_t bytesPerRow,
    CGColorSpaceRef cg_nullable space, CGBitmapInfo bitmapInfo,
    CGDataProviderRef cg_nullable provider,
    const CGFloat * __nullable decode, bool shouldInterpolate,
    CGColorRenderingIntent intent)

以及创建 CGContextRef 的时候

CGContextRef __nullable CGBitmapContextCreate(void * __nullable data,
    size_t width, size_t height, size_t bitsPerComponent, size_t bytesPerRow,
    CGColorSpaceRef cg_nullable space, uint32_t bitmapInfo)

bitmapInfo 参数都是 包含了两部分，而不是直接用 CGBitmapInfo 枚举值，这两部分就是 CGImageAlphaInfo & CGImageByteOrderInfo，前者用来描述 Alpha 通道，后者用来描述字节顺序。

通过二者的组合（使用 | 运算符），最终得到新的 UInt32 值来作为 BitmapInfo。

列举部分组合的效果

CGImageAlphaInfo

Alpha 通道的描述

typedef CF_ENUM(uint32_t, CGImageAlphaInfo) {
    kCGImageAlphaNone,               /* For example, RGB. */
    kCGImageAlphaPremultipliedLast,  /* For example, premultiplied RGBA */
    kCGImageAlphaPremultipliedFirst, /* For example, premultiplied ARGB */
    kCGImageAlphaLast,               /* For example, non-premultiplied RGBA */
    kCGImageAlphaFirst,              /* For example, non-premultiplied ARGB */
    kCGImageAlphaNoneSkipLast,       /* For example, RBGX. */
    kCGImageAlphaNoneSkipFirst,      /* For example, XRGB. */
    kCGImageAlphaOnly                /* No color data, alpha data only */
};

None，无：

  • kCGImageAlphaNone 没有 alpha 通道，就是只有 RGB

Premultiplied，预乘，会存 alpha 值：

  • kCGImageAlphaPremultipliedLast 有 alpha 通道，且在最末尾，会存储 alpha 值，且 alpha 值会与 RGB 预乘
  • kCGImageAlphaPremultipliedFirst 有 alpha 通道，且在最开头，会存储 alpha 值，且 alpha 值会与 RGB 预乘

不预乘，会存 alpha 值：

  • kCGImageAlphaLast 有 alpha 通道，且在最末尾，会存储 alpha 值，但是不预乘
  • kCGImageAlphaFirst 有 alpha 通道，且在最开头，会存储 alpha 值，但是不预乘

NoneSkip，不预乘，不存 alpha 值

  • kCGImageAlphaNoneSkipLast 有 alpha 通道，且在最末尾，但是 不存储 alpha 值
  • kCGImageAlphaNoneSkipFirst 有 alpha 通道，且在最末尾，但是 不存储 alpha 值

Only，只有 alpha 值

  • kCGImageAlphaOnly 只有 alpha 通道，没有 RGB

PS: 预乘，即对像素点设置颜色的时候，先将 alpha 值乘以 RGB 值。比如说对于一个像素颜色为 (250, 100, 50)，alpha 为 0.1，如果预乘，其像素颜色就为 (25, 10, 5)。优点是可以 提升性能，不用在解码、渲染的时候才去算最终颜色。

但是在 iOS 中，仅支持部分 AlphaInfo，可以参考 官方文档：

可以看出，kCGImageAlphaLast 和 kCGImageAlphaFirst 是不被支持的 。且官方也表示，最好使用 Premultiplied 的 AlphaInfo。

CGImageByteOrderInfo

typedef CF_ENUM(uint32_t, CGImageByteOrderInfo) {
    kCGImageByteOrderMask     = 0x7000,
    kCGImageByteOrderDefault  = (0 << 12),
    kCGImageByteOrder16Little = (1 << 12),
    kCGImageByteOrder32Little = (2 << 12),
    kCGImageByteOrder16Big    = (3 << 12),
    kCGImageByteOrder32Big    = (4 << 12)
} CG_AVAILABLE_STARTING(10.0, 2.0);
  • kCGImageByteOrderDefault iOS 是 32 位大端序
  • kCGImageByteOrder16Little 16 位小端序
  • kCGImageByteOrder32Little 32 位小端序
  • kCGImageByteOrder16Big 16 位大端序
  • kCGImageByteOrder32Big 32 位大端序

对于大端序而言，颜色排列是 RGB 顺序，如果是配合 alphaLast A 就会在末尾，变成 RGBA；

对于小端序而言，颜色排列是 BGR 倒序，如果是配合 alphaLast A 就会在 开头，变成 ABGR，因为是整段顺序调转，因此此处的 Last 实际上是我们理解的第一个。

16 位中，每 5 位表示 RGB，剩余 1 位用以表示 alpha；

32 位中，每 8 位表示一个颜色分量。

另外如果想做适配，可以使用 ByteOriderXXHost 宏：

#ifdef __BIG_ENDIAN__
# define kCGBitmapByteOrder16Host kCGBitmapByteOrder16Big
# define kCGBitmapByteOrder32Host kCGBitmapByteOrder32Big
#else    /* Little endian. */
# define kCGBitmapByteOrder16Host kCGBitmapByteOrder16Little
# define kCGBitmapByteOrder32Host kCGBitmapByteOrder32Little
#endif

iOS 系统是 小端序，因此 kCGBitmapByteOrder32Host == kCGBitmapByteOrder32Little

另外，如果需要将 UInt32 从 大/小 端序转到系统的字节序，可以使用下述宏：

uint32_t CFSwapInt32BigToHost(uint32_t arg)
uint32_t CFSwapInt32LittleToHost(uint32_t arg)

同样的，如果需要转化 UInt16 或者 UInt64，将上面宏 32 改成对应值即可，具体可以自行查阅 CoreFoundation/CFByteOrder.h 头文件了解。

总结

使用 AlphaInfo 和 ByteOrderInfo 组合生成颜色元素顺序的时候，可以分成如下步骤：

设初始排序为 RGB

  • 1、先通过 AlphaInfo 找到 A 的位置

    • First 则 ARGB
    • Last 则 RGBA

  • 2、通过 ByteOrderInfo 确定是否要翻转排序

    • Big 则无需翻转
    • Little 则 整体翻转，比如 ARGB -> BGRA
