iOS 的 KVO & KVC 本质

 Posted on January 1, 0001  |  1 minutes  |  189 words  |  Kim

  • KVO：监听属性的变化

  • KVC：属性（成员变量）设值

KVO

KVO 的全称 Key-Value Observing，俗称 “键值监听”，可以用于监听某个对象属性值的改变。

KVO 的底层实现

设有对象:

MyObject *obj = [MyObject new];

当 MyObject 的实例对象 obj 调用了 addObserver: 的方法之后，发生了如下的事：

  • 1、 通过 Runtime 动态生成一个新类对象 NSKVONotifying_MyObject，该类对象 继承自 MyObject

  • 2、 将 obj 的 isa 指针，指向 NSKVONotifying_MyObject 类对象 （isa-swizzling）

  • 3、 新的类对象重写父类对应被监听属性的 set 方法，会依次调用下述方法：

    • willChangeValueForKey 方法
    • 父类的 setter 方法实现
    • didChangeValueForKey 方法

    而 did(will)ChangeValueForKey 方法内部又会调用监听器的observeValueForKeyPath:ofObject:change:context: 监听方法。

关闭默认触发的 KVO

在如下方法返回 NO 即可。

//默认YES（自动）
+(BOOL)automaticallyNotifiesObserversForKey:(NSString *)key{
    return NO;
}

如何手动触发 KVO？

手动调用 willChangeValueForKey: 和 didChangeValueForKey:

通过KVC修改属性会触发KVO么？

会！

因为 KVC 的原理是会找到 setParam: 的方法，然后调用之

KVO 崩溃的原因？怎么预防

原因有如下：

  • 1、KVO 添加次数和移除次数不匹配

    • 移除了未注册的 Observer
    • 重复移除

  • 2、被观察的对象提前被释放，但是它仍然被注册着 KVO

  • 3、添加了观察者，但观察者未实现 observeValueForKeyPath:ofObject:change:context: 方法

  • 4、添加或者移除时 keypath == nil，导致崩溃

如何预防？

可以有如下的步骤：

    1. 创建一个类 （MyKVO），使用其单例进行 KVO 的分发
    1. 置换系统的 添加 & 移除 KVO 的方法，让上述单例去实现
    1. 该单例使用 字典 去管理，KVO 的观察者、被观察者、Object 等，然后在其内部进行 KVO 的方法，保证添加和移除的唯一性，并且做 nil 的保护
    1. 方法置换 dealloc 到新的方法，保证对象释放的时候能把上述单例管理的该对象给移除掉。

KVO 各个参数的作用？

/**
 1. self.person：要监听的对象
 2. 参数说明：
    * @param addObserver  观察者，负责处理监听事件的对象
    * @param forKeyPath 要监听的属性
    * @param  options 观察的选项（观察新、旧值，也可以都观察）
    * @param context 上下文，用于传递数据，可以利用上下文区分不同的监听
 */
[self.person addObserver:self forKeyPath:@"name" options:NSKeyValueObservingOptionNew | NSKeyValueObservingOptionOld context:@"Person Name"];

/**
 *  当监控的某个属性的值改变了就会调用
 *
 *  @param keyPath 监听的属性名
 *  @param object  属性所属的对象
 *  @param change  属性的修改情况（属性原来的值`oldValue`、属性最新的值`newValue`）
 *  @param context 传递的上下文数据，与监听的时候传递的一致，可以利用上下文区分不同的监听
 */
- (void)observeValueForKeyPath:(NSString *)keyPath ofObject:(id)object change:(NSDictionary *)change context:(void *)context
{
    NSLog(@"%@对象的%@属性改变了：%@", object, keyPath, change);
}

其中的 context 参数可以传入不同的参数，以区别是哪儿触发的。
