编译 & 加载 原理

我们所写的代码是一个 XCode 的工程，在按下 command + r 之时，会将其 编译 成 MACH-O 可执行文件；然后放到手机上，再点击应用的时候，系统的内核会开启一个进程，然后通过 dyld 执行 加载 流程。

一、前驱

iOS 是一门动态语言，动态主要是通过 objc/runtime.h 库实现，我们的函数调用通过 id objc_msgSend(id self, SEL op, ...) 函数发送消息去找到对应的函数实现，然后调用之。

SEL 是函数名，指向函数的实现（IMP）

IMP 是函数的实现，是个函数指针，指向一个函数。

1. 动态库共享缓存区 (dyld shared cache)

在我们 iOS 系统中，会有这么一块儿区域，存放着公用的动态库，比如说 UIKit 、Foundation 等等的 系统库。

注意，每次重启手机，都会懒加载这些库，因此其地址不是固定的，也是 动态 的。

2. 符号表 （symbols）

我们的工程在 编译期 会预先在 MACH-O 可执行文件中留出 _DATA 段，来当做符号表。

在 编译期，会找到所有我们工程引用的上述 动态库共享缓存区 的方法，并且将其设置成一个指针，此时指向的地址为 NULL，然后存放到这个符号表中。

只有在 运行时，dyld 才将 动态库共享缓存区 中的系统函数 绑定 到我们的 符号表 中，这样我们就能通过该表找到系统函数对应的函数指针的地址，并且对其进行调用。

上述的过程被称为 PIC（位置代码独立. Position Independent Code） 技术。

可以说，系统库函数也是 动态 的，因此也能通过 fishhook 等库去对系统函数进行 hook（实际上是对符号的重新绑定的过程）。

3. 动态链接器 dyld （The dynamic link editor）

在应用被编译成 MACH-O 可执行文件之后，通过 dyld 执行 链接 & 加载。

4. 随机地址技术 （ALSR）

iOS 系统会对存储器的地址进行一段随机的偏移，即：

真实的物理地址 = 相对地址 + ALSR

相对于固定地址，更能防止攻击者直接获取函数的地址获取函数或者变量的地址进行攻击。

二、dyld 的加载流程

  • 1、 _dyld_start

    调用时机：点击应用的时候，系统的内核会开启一个进程，然后通过 dyld 执行 加载 流程

    作用：调用下一步的函数

  • 2、 dyldbootstrap :: start

    • a. 计算 ALSR

    • b. 调用下一步的函数

  • 3、dyld::_main

    • a. 检查当前环境 & 获取架构

    • b. 加载 动态库共享缓存区 的内容

    • c. 实例化主程序，会进行 应用签名

    • d. 加载 & 插入 动态库

    • e. 链接主程序，对 系统库、动态库 进行 符号绑定

    • f. _objc_init 初始化，调用所有类的 +load 方法

    • g. doInitialization 调用 c++ 的构造方法（constructor）

    • h. 找到并返回 主程序的 main 函数（主程序入口）。

三、编译

iOS 的编译分两种情况，OC 使用 Clang 作为 编译器前端，而 Swift 使用 swift()，但是两种语言的编译后端都是 LLVM。

整体的流程：

Clang (swift()) --> LLVM Optimizer --> LLVM Generator

1. 编译器前端 Clang(swift())

进行 语法分析，语义分析，生成中间代码 ( intermediate representation ) 。在这个过程中，会进行类型检查，如果发现错误或者警告会标注出来在哪一行。

2. 编译器后端 LLVM

  • 优化 （Optimizer）

    进行 BitCode 的生成，链接期优化等等

  • 生成器 （Generator）

    针对不同的架构，比如 arm64 等生成不同的机器码
