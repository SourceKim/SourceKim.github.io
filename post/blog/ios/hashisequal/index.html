在 OC 中：

  • hash 的类型是 NSUInteger，可以初步用来判等

  • isEqual 是相等的绝对判定

  • == 仅是对比变量地址的指针是否相等

一、 == & isEqual:

由于 OC 不能像 C++ 一样重载运算符，因此他们的含义是不一样的：

  • == 是对比对象的地址（指针）是否相等

  • isEqual: 是对比对象的值是否相等

二、 isEqual: & hash

这两个都是 NSObject 协议的内容：

- (BOOL)isEqual:(id)object;
@property (readonly) NSUInteger hash;

  • ①、 如果两个对象 isEqual: 返回 true，代表他们的 hash 一定相同

  • ②、 但是如果 hash 相同，isEqual: 则不一定返回 true

但是为什么呢？

1. 对于定律 ①

我们一般这么定义 isEqual:

假设有个类 MyClass

- (BOOL)isEqual:(id)object {
    if (self == object) {
        return YES;
    }
    
    if (![object isKindOfClass: [MyClass class]]) {
        return NO;
    }
    
    return [self isAllPropertiesEqualTo:(MyClass *)object];
}

会先判断 地址 是否相同 （直接用 ==

然后判断是否 同类，否则直接返回 false

然后再对比各自的 属性 是否一致

又因为，系统一般直接返回 自身内存地址 定义 hash，所以在第一步的时候就注定他们的 hash 是一致的。

2. 对于定律 ②

由于 hash 冲突，可能会导致两个对象的 hash 一致，造成 hash 冲突的原因目前我知道 2 个：

  • a、系统的 hash 负载因子 过大，容易冲突

  • b、NSString 自身缺陷

引用 CFString.h 的一段话

/* String hashing: Should give the same results whatever the encoding; so we hash UniChars.
If the length is less than or equal to 96, then the hash function is simply the 
following (n is the nth UniChar character, starting from 0):
   
  hash(-1) = length
  hash(n) = hash(n-1) * 257 + unichar(n);
  Hash = hash(length-1) * ((length & 31) + 1)

If the length is greater than 96, then the above algorithm applies to 
characters 0..31, (length/2)-16..(length/2)+15, and length-32..length-1, inclusive;
thus the first, middle, and last 32 characters.

Note that the loops below are unrolled; and: 257^2 = 66049; 257^3 = 16974593; 257^4 = 4362470401;  67503105 is 257^4 - 256^4
If hashcode is changed from UInt32 to something else, this last piece needs to be readjusted.  
!!! We haven't updated for LP64 yet

NOTE: The hash algorithm used to be duplicated in CF and Foundation; but now it should only be in the four functions below.

Hash function was changed between Panther and Tiger, and Tiger and Leopard.

大意就是 这个字符串的大小如果 <=96，则保证 hash 的安全；如果大小 >96 了，则无法保证安全，它只会对前 32，中 32，后 32 进行 hash

因此会加大 hash 冲突的概率。

3. 既然 hash 不稳定，那为什么还要有 hash？

因为查找的时候 hash 方法快，时间复杂度是 O(1)

三、重写 hash 的最佳实践：

对对象的 各个属性 进行 XOR

- (NSUInteger)hash {
    return [self.name hash] ^ [self.birthday hash];
}

四、 Dictionary & Set 的存取

这两个数据结构底层都是 哈希表 实现的

取

在取值的时候，会根据 hash 方法去取得对应的值，但是由于 hash 不稳定 的情况，因此有的时候底层调用还会判定 isEqual: 的函数

存

  • 由于有动态的数据结构， 即可变（mutalble），因此会对 哈希表 进行 动态扩容，主要有如下方法：

      1. 拉链法 （红黑树与链表的切换）
      1. 开放地址线性探测法 （对比插入成功 & 查找成功的值，返回新值）

  • NSMutalbleDictionary 添加元素的时候，深拷贝 key，会有新生成的 hash

    NSMutableDictionary *dict = [NSMutableDictionary dictionary];
dict setObject:<#(nonnull id)#> forKey:<#(nonnull id<NSCopying>)#>

    可以看到 key 不仅仅要是 NSObject，还要实现 NSCopying 协议，因为内部会进行 深拷贝，防止原来的对象改变之后，使用原对象在字典中找不到对应的 value。
