TCP 的拥塞控制 （滑动窗口协议）

 发表于 July 28, 2019  |  1 分钟  |  148 个字  |  Kim
  • 名词解释：
  • 策略：
    • 1. 慢启动
    • 2. 拥塞避免
    • 3. 快重传 （防止因为丢包误会了是网络拥堵，导致窗口变小）
    • 4. 快恢复
  • 总结

  • TCP 的拥塞控制：通过对发送端的 流量控制，即控制其报文发送速度，使之能匹配接收端的速度，达到防止网络性能下降的策略。

  • 滑动窗口协议：通过该协议来实现 TCP 的拥塞控制。

名词解释：

cwnd（congestion window） 拥堵控制窗口，初始值为 1

swnd（send window） 发送窗口，代表每次可以发送的数据量

ssthresh（slow start threadshold） 慢启动阈值，一般为 cwnd / 2

策略：

1. 慢启动

此时 每收到一个 ACK 的时候，cwnd + 1

即当这次我发送了 4 个，回来 4 个 ACK，此时我的 cwnd = 8，下次我能发送 8 个数据

当 cwnd > ssthresh 的时候，采用下述 拥塞避免 策略

2. 拥塞避免

每个轮次回来时，cwnd + 1

假设我现在的 cwnd = 100，超了 ssthresh = 60，发送了 100 个数据，回来了 100 个 ACK，但是我此时的 cwnd = 101，因为是 每轮 +1，而不是个。

假设此时发生了 超时重传：

  • 降低阈值 ssthresh /= 2
  • 重置拥塞窗口 cwnd = 1
  • 然后重新进行 慢启动

3. 快重传 （防止因为丢包误会了是网络拥堵，导致窗口变小）

意思是让 发送方尽快重传（而不是等到重传计时器超时再重传），此处的尽快的意思是 发送方收到了下述 2 中的 三个 重复确认 就立即重传

此时，对接收方有两个要求：

    1. 要求 接收方要立即发送确认 ACK（而不是等到自己发送数据时候才带上 ACK）
    1. 即使收到了序号不对的数据，也要立即发出对已经收到的数据进行 重复确认！

同时，在进行重传的时候，要同时使用下述 快恢复

4. 快恢复

在收到 3 个 重复确认 的时候，要使用如下算法进行快恢复：

  • 更新阈值 ssthresh /= 2 or ssthresh += 3
  • 更新拥塞窗口 cwnd = ssthresh
  • 使用拥塞避免

总结

  • 慢启动 使用时机： 刚启动 or 发生超时重传调整之后

  • 拥塞避免 使用时机： 慢启动时 cwnd > ssthresh or 快恢复调整之后

  • 快重传 使用时机： 收到 3 个 重复确认

  • 快恢复 使用时机： 收到 3 个 重复确认
