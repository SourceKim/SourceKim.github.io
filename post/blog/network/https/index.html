HTTPS = HTTP + SSL/TLS 协议

SSL 的全称是 Secure Sockets Layer，即安全套接层协议，是为网络通信提供安全及数据完整性的一种安全协议。

TLS 的全称是 Transport Layer Security，即安全传输层协议。

即 HTTPS 是安全的 HTTP。

连接过程

  1. 客户端 -> 服务器 【 访问 https 】

    客户端生成 随机数 C

    此时会发送： TLS 版本、支持的加密算法列表

  2. 服务器 -> 客户端 【 颁发证书 】

    此时会返回： 公钥、数字证书、随机数 S

  3. 客户端 【 验证证书 】

  4. 客户端 【 组装会话秘钥 】

    使用服务器的 公钥 + C 生成 前主秘钥（PMS），

    使用 PMS + C + S 组装成 会话秘钥

  5. 客户端 -> 服务器 【 发送 PMS 】

    客户端发送 PMS 给服务器

  6. 服务器 【 组装 会话秘钥 】

    通过 私钥 解开 PMS

    然后和 步骤 4 一样： 使用 PMS + C + S 组装成 会话秘钥

  7. 两段通过 会话秘钥 进行通讯。

总结：

在 会话秘钥 组装之前，使用 非对称加密，这样可以保证安全 （握手阶段）；

在 会话秘钥 组成之后，使用 对称加密，这样可以保证通讯的效率 （通讯阶段）。

PS：非对称加密 耗时大大高于 对称加密，同时，安全性也比其强。

可否完全防御 中间人攻击？

不可以，因为 证书可以造假。

中间人攻击 是什么？

正常情况下浏览器与服务器在 TLS 链接下内容是加密的，第三方即使可以嗅探到所有的数据，也不能解密。 中间人可以与你建立链接，然后中间人再与服务器建立链接，转发你们之间的内容。这时候中间人就获得了明文的信息。

为什么不可以完全防御？

因为我们一般通过 证书 去判断 公钥 是否是该网址的，但是如果有些机构的证书不可信（不知名机构颁发或者自签名）的话，不容易发觉是伪造的。

此时我们无法判断出这次握手的 公钥沟通流程 是和 真网址 还是 假网址（中间人） 进行。后者通过转发消息，即可完成嗅探。

如何防范？

在访问https链接的时候，查看一下服务器提供的证书是不是正确的。

因为除非入侵并取得服务器的证书私钥，否则中间人是不能完全伪装成服务器的样子的。
