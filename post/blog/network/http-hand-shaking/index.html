HTTP 三次握手 & 四次挥手，以及一些常规概念的 FAQ。

一、流程

什么是握手？

握手、挥手的意思就是 发送数据包。

那么三次握手，含义就是：建立 HTTP 连接的时候，客户端和服务端共计需要发送 3 个包，才能完成连接；

同理四次挥手，含义就是：断开 HTTP 连接的时候，客户端和服务端共计需要发送 4 个包，才算断开连接。

在此之前，需要理解如下的一些概念，

先是标志：

  • SYN Synchronous 建立同步连接标志

  • ACK Acknowledgement 确认标志

  • FIN Finish 完成标志

再是一些序号

  • seq Sequence number 顺序号

  • ACKnum Acknowledge number 确认码

还有一个概念

  • MSL Maximum Segment Lifetime 报文最大生存时间，超时之后这个包就会被丢弃

三次握手流程

  • 1、客户端 -> 服务器 （SYN=1，seq=x）

    客户端请求连接，因此 SYN=1。

    客户端生成随机序号 x，发送给服务器。

  • 2、服务器 -> 客户端 （SYN=1，ACK=1，seq=y，ACKnum=x+1）

    服务端请求连接，因此 SYN=1；同时服务器回应客户端连接，因此 ACK=1。

    服务端生成随机序号 y，以及将客户端生成的 x+1 返回给客户端

  • 3、客户端 -> 服务器 （ACK=1，ACKnum=y+1）

    客户端回应服务器连接，因此 ACK=1。

    客户端将服务端生成的随机序号 y，加 1 返回给服务端，即 ACKnum=y+1.

四次挥手流程

  • 1、客户端 -> 服务器 （FIN=1，seq=x）

    客户端发起关闭连接请求，因此 FIN=1。

    客户端生成随机序列号 x，发送给服务器。

  • 2、服务器 -> 客户端 （ACK=1，ACKnum=x+1）

    服务器响应客户端的关闭请求，因此 ACK=1。

    服务器将客户端生成的随机序列号 x，加 1 返回给客户端，即 ACKnum=x+1。

  • 3、服务器 -> 客户端 （FIN=1，seq=x）

    服务器发起关闭连接请求，因此 FIN=1。

    服务器生成随机序列号 y，发送给客户端。

  • 4、客户端 -> 服务器 （ACK=1，ACKnum=y+1）

    客户端响应服务器的关闭请求，因此 ACK=1。

    客户端将服务器生成的随机序列号 y，加 1 返回给客户端，即 ACKnum=y+1。

二、细节问题 FAQ

这里整理一些常见的面试问题：

1、为什么需要三次握手？两次不可以吗？

首先需要了解 TCP 的 失效 请求连接概念：客户端发送 SYN 请求之后，会在等待 ACK 超时之后，重新发送新的 SYN 连接请求，此时旧的 SYN 就是 失效连接。

TCP 的三次握手是为了防止服务端接收到失效连接，造成错误的响应。

两次握手由于失效连接出错的情景：

客户端由于网速慢，第一个请求超时了，此时发送了第二个连接请求，在这个连接请求被正确处理了之后，第一个连接请求才到达了服务器，此时服务器会一直处于 ESTABLISH 状态，等待客户端的数据，造成资源浪费，也因此会堵塞下一个连接请求。

因此我们需要第三次握手，来让客户端确认收到服务器的连接（通过 ACKnum），假如未被确认则丢弃该次连接请求。

2、为什么握手是三次，而挥手需要四次？

因为在四次挥手的过程中，服务器在第一次响应的时候仅仅是告诉客户端收到了其断开的请求，但是此时 服务器可能还有未发送完的数据，要等待数据都发送完了之后，服务器才会向客户端发送断开的请求。

如果服务器直接将 ACK 和 FIN 一起发送，就会导致服务器的数据未发送完就断开了两端的连接。

3、为什么客户端在发送完断开连接 ACK 后要等待 2MSL？

首先需要了解，当 FIN 释放连接请求等待 ACK 超时会重新发送 FIN。

客户端在发送了 ACK 之后等待 2MSL 是为了防止客户端的回应断开连接的包丢失，导致服务端永远无法正常关闭。

假如客户端发送完 ACK 就直接 CLOSED 了，此时该包丢失了，服务器重新发送的 FIN 就会永远得不到 ACK （因为客户端 CLOSED 了），因此还需要等待一下，看服务端时候会因为超时而重新发送 FIN。
